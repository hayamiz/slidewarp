# 技術スタック検討ドキュメント

学会撮影スライド写真の一括処理コマンド（以下 `slidewarp`）の実装技術スタックを検討する。

---

## 0. 現在の採用状況（本文は検討経緯として保存）

> **本節（§0）が現在の実採用の正である。以降の §1〜§6 は選定検討時の記録であり、
> §6 の「未決定事項」を含め、当時未確定だった論点はすべて §0 で解決済みとして読むこと。**
> 各節の推奨・評価は検討当時の判断であり、必ずしも実採用と一致しない（差異は本節で明示する）。

### 採用したスタック
- **本体は純Rust（`image` + `imageproc`）、OpenCV 非依存の単一バイナリ**。
  `Cargo.toml` の依存は `image` / `imageproc` / `rayon` / `clap` / `serde` /
  `serde_json` / `anyhow` / `walkdir` / `mimalloc` のみで、**`opencv` crate は不使用**。
  配布は musl ターゲットで真のシングルバイナリ（静的リンク）。
  → §4 の「本命=選択肢B（Rust + `opencv` crate）」は**検討当時の推奨**であり、
    実際には採らなかった。実採用は**選択肢D 寄りの構成**（純Rust）に相当する
    （ただし後述のとおり ML の `ort` は用いていない）。
- **ML は未導入**: `ort`（ONNX Runtime）は依存に含まれない。ML セグメンテーションには
  進まず、classical な幾何処理を自作した（自前 Douglas-Peucker、`imageproc` の Hough /
  warp、上下辺リファイン等）。
  → §3(d)/選択肢D が「工数最大・品質リスク最大」と評価した自作路線は、実際には
    開発時の手元 24 枚で人手評価ほぼ全て良好まで到達している（詳細は `CLAUDE.md` の「現状」）。
- **Python（`python/`）は実験用**として併存する。認識アルゴリズムは Rust 本体が正であり、
  Rust 未移植の `--remove-people`（人物セグメンテーション + inpaint、torchvision DeepLabV3）
  のみ Python 側に実装がある。

### §6「未決定事項」の解決
- **主軸スタック** → 純Rust（`image` + `imageproc`）に決定（Python は実験用に併存）。
- **ML を初期から入れるか** → classical で開始し、`ort` は未導入（人物除去のみ Python 側の
  torch 実装）。
- **検出失敗時のポリシー** → 決定済み。低信頼時は `--on-low-confidence`（既定はスキップ、
  原本コピーも選択可）でレビュー用に振り分け、`report.html` を生成して人手確認する。
  誤補正画像をバッチで量産しないことを最優先の安全策とする。
- **出力形式・命名・サイドカー** → 出力アスペクトは 4:3 か 16:9 にスナップし、
  `report.html` を既定生成する（`--no-report` で抑止）。細目（命名規則・JSON サイドカーの
  要否）は実装準拠とし、本節では断定しない。

## 1. 要件の整理

### 機能要件
- 入力: 1つ以上の写真ファイル、または写真が入ったフォルダ（再帰探索）
- 出力: 指定した出力ディレクトリに処理済み画像を書き出す
- 処理パイプライン:
  1. 写真中の「スライドが映っていると推定される矩形領域」の検出
  2. トリミング + 台形補正（透視変換）
  3. シャープ化
  4. **optional**: 露出自動補正、色調補正（ホワイトバランス等）
- 頑健性: スライドが画角からはみ出している / 人影・話者が被っている場合でも、
  ある程度「矩形らしい領域」を推定できること

### 非機能要件（優先度順）
1. **画像処理性能が最優先**（ライブラリインストールが必要でも可）
2. シングルバイナリ配布ができれば望ましい（が、性能とトレードオフなら妥協可）
3. Ubuntu 24.04 で動作すること（必須）
4. macOS でも動くとベター

> サンプル写真（`input-samples/`）はいずれも 1600x1200 (4:3)。実会場では暗所・
> 逆光・強い台形歪み・観客の頭による下辺のオクルージョンが頻出することを確認済み。

## 2. 中核となる CV アルゴリズムと、それがスタック選定を規定する理由

「どの言語か」よりも先に「**どの画像処理・CVライブラリの機能が使えるか**」が
実装の成否を決める。スライド検出で必要になる要素技術は以下:

| 要素技術 | 用途 | OpenCV | 純Rust/純Go |
|---|---|---|---|
| Canny / Sobel エッジ | 輪郭抽出 | ○ | △ (imageproc) |
| findContours + approxPolyDP | 四角形候補の抽出 | ○ | ✕ (要自作) |
| Hough 直線検出 | 辺の直線検出（オクルージョン対応の要） | ○ | △ (imageproc) |
| getPerspectiveTransform / warpPerspective | 台形補正 | ○ | ✕ (要自作) |
| CLAHE / ヒストグラム | 露出補正 | ○ | △ |
| DNN 推論 (ONNX) | ML によるセグメンテーション（最も頑健） | ○ (dnn) / ort | ort (Rust) |

**結論の先取り**: オクルージョン・画角はみ出しへの頑健性を現実的な工数で得るには、
`findContours` / Hough / 透視変換が揃った **OpenCV 相当のライブラリはほぼ必須**。
純正言語ライブラリでフルスクラッチすると工数が跳ね上がる。したがって選定は
「OpenCV をどの言語から、どう配布するか」の問題に帰着する。

## 3. 選択肢の比較

### 選択肢A: Python + OpenCV（+ 任意で ONNX モデル）★開発速度最優先
- **CVライブラリ**: `opencv-python`, `scikit-image`, `numpy`, `Pillow`
- **性能**: OpenCV 本体はネイティブ C++ なので画像処理は十分高速。バッチは
  `multiprocessing` で並列化。ボトルネックはほぼ OpenCV 内で処理される。
- **配布**:
  - `uv` / `pip` + venv（研究室内利用ならこれで十分）
  - PyInstaller / Nuitka で「1ディレクトリ or 1ファイル」化は可能だが真の
    シングルバイナリではなく、OpenCV/numpy を同梱した数百MBのバンドルになる
  - Docker イメージ配布（Ubuntu 24.04 なら最も確実）
- **頑健性**: 最強。`cv2.dnn` や `onnxruntime` で ML セグメンテーション/文書検出
  モデルを容易に追加でき、classical 手法との併用が最短。scikit-image で補正も豊富。
- **クロスプラットフォーム**: pip で Ubuntu/macOS とも wheel が用意され導入容易。
- **短所**: シングルバイナリ要件を最も満たしにくい。実行時 Python 環境が必要。
- **向き**: アルゴリズムを素早く試行錯誤したい / 研究室内配布が主目的。

### 選択肢B: Rust + `opencv` crate ★配布性と性能のバランス
- **CVライブラリ**: [`opencv`](https://crates.io/crates/opencv) crate（OpenCV C++ への bindings）,
  補正用に `image` / `imageproc` を併用可
- **性能**: OpenCV ネイティブ + Rust の薄いラッパで最速級。`rayon` で容易に並列化。
- **配布**:
  - `cargo build --release` で自前コードは単一バイナリ化されるが、**OpenCV 共有
    ライブラリ (`libopencv_*.so`) への動的リンクが残る**（真の単一バイナリではない）
  - vcpkg で OpenCV を static build して静的リンクすれば単一バイナリ化も理論上可能
    だが、ビルド環境構築が重くハマりどころが多い（実務では動的リンク + apt 導入が現実的）
- **頑健性**: OpenCV フル機能 + `ort` crate で ONNX 推論も追加可能。Python と同等。
- **クロスプラットフォーム**: Ubuntu は apt (`libopencv-dev`)、macOS は Homebrew。
  ビルド時に OpenCV のバージョン差でつまずくことがある。
- **短所**: OpenCV crate のビルド/リンク設定が初手のハードル。開発イテレーションは
  Python より遅い。
- **向き**: 性能を保ちつつ配布しやすい CLI を作りたい。

### 選択肢C: Go + `gocv` ★CLI 配布運用が楽
- **CVライブラリ**: [`gocv`](https://gocv.io/)（OpenCV bindings）
- **性能**: OpenCV ネイティブなので良好。並列化も goroutine で容易。
- **配布**: Go は本来単一バイナリが強みだが、gocv は cgo 経由で OpenCV 共有ライブラリ
  に依存するため**真の単一バイナリにはならない**（Rust/opencv と同じ制約）。
- **頑健性**: OpenCV 機能は使えるが、ONNX/DNN 連携は Python/Rust よりエコシステムが薄い。
- **短所**: cgo により Go の可搬性の利点が減殺される。CV コミュニティも小さい。
- **向き**: 既に Go 資産がある場合。今回は積極的な理由が薄い。

### 選択肢D: 純 Rust（`image` + `imageproc` + `ort`）★真の単一バイナリ最優先
- **CVライブラリ**: `image`, `imageproc`, `ndarray`, ML は `ort`(ONNX Runtime)
- **配布**: musl ターゲットで**真のシングルバイナリ**（依存ライブラリなし）が作れる。
  最も配布要件に忠実。
- **性能**: 良好だが、`findContours` / `warpPerspective` 相当は自前実装が必要で、
  最適化されていない分 OpenCV に劣る場合がある。
- **頑健性/工数**: 四角形抽出・透視変換・幾何補正をほぼ自作。工数とバグリスクが最大。
  ただし ML 路線（`ort` でセグメンテーション → マスクから四角形フィット）に振り切れば、
  classical 幾何処理の自作を減らせる。
- **短所**: 実装工数・アルゴリズム品質のリスクが最も高い。
- **向き**: 「依存ゼロの単一バイナリ」が絶対要件のとき。今回は優先度2なので過剰。

### （参考）C++ + OpenCV
性能は最大だが、開発生産性・メモリ安全性・ビルドの手間で Rust/Python に劣り、
今回の優先度（性能は OpenCV 使用でどの言語でも頭打ち）では選ぶ積極的理由が乏しい。

## 4. 推奨

> ※ 本節の推奨は検討当時のもの。実採用は §0 を参照（本命だった選択肢B は不採用、純Rust を採用）。

優先度が「①性能 > ②単一バイナリ > ③Ubuntu > ④macOS」で、かつ
**「単一バイナリは性能と引き換えなら妥協可」**と明言されている点を重視する。

- **本命: 選択肢B（Rust + `opencv` crate）**
  - 性能は OpenCV ネイティブで最上位。`rayon` によるバッチ並列も容易。
  - 自前コードは単一バイナリ化でき、配布は「バイナリ + `apt install libopencv-dev`」
    または Docker で完結。真の単一バイナリではないが要件上許容範囲。
  - CLI としての堅牢性・型安全性・エラー処理が Python より優れ、長期運用に向く。
  - 頑健性が足りなければ `ort` で ML モデルを後付けできる拡張余地がある。

- **対抗: 選択肢A（Python + OpenCV）**
  - アルゴリズムの試行錯誤速度は圧倒的。まず Python でパイプラインと検出ロジックを
    確立し、必要なら Rust へ移植、という二段構えも合理的。
  - 「研究室内で自分/共同研究者が使う」用途なら Python のままでも実用上十分。

### 推奨する進め方（ハイブリッド）
1. **Python + OpenCV でアルゴリズムをプロトタイプ**し、`input-samples/` で検出精度を検証
2. 検出/補正ロジックが固まったら、配布要件に応じて
   - そのまま Python 配布（uv / Docker）で確定、または
   - Rust + `opencv` crate へ移植して単一バイナリ寄りの配布に
3. 頑健性が classical 手法で不足する場合のみ ML セグメンテーションを追加

> どちらを主軸にするかで初期スキャフォールドが変わるため、着手前に方針を確認する。

## 5. スライド検出アルゴリズム設計（言語非依存）

頑健性要件（はみ出し・オクルージョン）を満たすため、**多段フォールバック**構成とする。

```
1. 前処理: 縮小（高速化）→ グレースケール → 平滑化
2. 明度事前分布: スライドは「暗い会場中の明るい矩形」→ 明度で screen 領域を大まかに分離
3. 候補生成（複数手法を並行し、スコアで統合）:
   (a) Canny エッジ + モルフォロジー閉 → findContours → approxPolyDP で四角形候補
   (b) Hough 直線検出 → 支配的な4直線の交点で四隅を復元
       ★ はみ出し・オクルージョンに強い（辺の一部が見えれば交点は画角外でも外挿可能）
   (c) 明度マスク → 凸包 / minAreaRect → 四角形フィット
   (d) optional: ONNX セグメンテーションモデルでスクリーン領域マスク → 四角形フィット
4. 候補スコアリング: 面積・矩形度・アスペクト比(4:3/16:9 に近いか)・内部明度・エッジ整合
5. 台形補正: 推定四隅 → getPerspectiveTransform → warpPerspective
   出力アスペクト比は消失点から真のアスペクトを推定、失敗時は 4:3/16:9 にスナップ
6. 強調: アンシャープマスクでシャープ化
   optional: CLAHE/ヒストグラム伸張で露出補正、gray-world 等でホワイトバランス
7. 書き出し（メタ情報: 元ファイル名・検出信頼度をログ/サイドカーに）
```

- (b) の直線交点方式が「画角はみ出し・下辺オクルージョン」への主対策。
- 候補が信頼度しきい値に満たない場合は **未処理コピー or スキップ + 警告**とし、
  誤検出で壊れた出力を出さない（バッチ運用での事故防止）。
